# 2-1 从找小姐姐买咖啡理解消息中间件

中间件（Middleware），是提供软件和软件之间连接的软件，以便于软件各部件之间的沟通。

“微信”就是中间件，而且是个消息中间件。

以下是订单系统各微服务的交互流程图：

```mermaid
graph TD
    A[用户] <--> B[订单微服务]
    B <--> C[积分微服务]
    B <--> D[骑手微服务]
    D <--> E[骑手]
    B <--> F[商家微服务]
    F <--> G[商家]
    B --> H[结算微服务]
    H <--> I[银行]
```

## 业务流程（同步直接调用）

```mermaid
sequenceDiagram
    participant U as 用户
    participant O as 订单微服务
    participant M as 商家微服务
    participant R as 骑手微服务
    participant S as 结算微服务
    participant P as 积分微服务

    %% 核心同步流程（无分支判断，仅展示主线）
    U->>O: 下单
    O->>M: 调用商家微服务（校验商品信息）
    M->>O: 响应订单微服务（商家OK）
    O->>R: 调用骑手微服务（分配骑手）
    R->>O: 响应订单微服务（骑手OK）
    O->>S: 调用结算微服务（与银行结算）
    S->>O: 响应订单微服务（结算OK）
    O->>P: 调用积分微服务（结算积分）
    P->>O: 响应订单微服务（积分OK）
    O->>U: 响应用户（下单成功）
```

### 同步直接调用的问题

- 业务调用链过长，用户等待时间长
- 部分组件故障会瘫痪整个业务
- 业务高峰期没有缓冲

## 业务流程（异步直接调用）

```mermaid
sequenceDiagram
    participant U as 用户
    participant O as 订单微服务
    participant M as 商家微服务
    participant R as 骑手微服务
    participant S as 结算微服务
    participant P as 积分微服务

    %% 异步响应阶段
    U->>O: 下单
    O->>O: 新建异步线程
    O-->>U: 响应用户（订单处理中）
    
    %% 异步线程内的后续同步调用（仅展示主线流程，无par/alt）
    O->>M: 调用商家微服务（校验商品信息）
    M-->>O: 响应订单微服务（商家OK）
    O->>R: 调用骑手微服务（分配骑手）
    R-->>O: 响应订单微服务（骑手OK）
    O->>S: 调用结算微服务（与银行结算）
    S-->>O: 响应订单微服务（结算OK）
    O->>P: 调用积分微服务（结算积分）
    P-->>O: 响应订单微服务（积分OK）
    O->>O: 更新订单状态
```

### 异步直接调用的问题

- ~~业务调用链过长，用户等待时间长~~ ✅
- ~~部分组件故障会瘫痪整个业务~~ ✅
- ~~业务高峰期没有缓冲~~ ✅
- 业务高峰期时产生大量的异步线程，造成线程池不够用或者内存爆满

### 使用消息中间件的优势

- 业务调用链短，用户等待时间短
- 部分组件故障不会瘫痪整个业务
- 业务高峰期有缓冲
- 业务高峰期时不会产生大量的异步线程

### 使用消息中间件的作用

- 异步处理
- 系统解耦
- 流量削峰和流控
- 消息广播
- 消息收集
- 最终一致性
